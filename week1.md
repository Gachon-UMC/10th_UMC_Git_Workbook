#  🌐 Week1 Git & GitHub 정리

---

##  1. Git이란?

Git은 파일의 변경 사항을 추적하고 여러 사용자 간 작업을 조율할 수 있도록 도와주는 **분산 버전 관리 시스템**이다.

즉, 파일의 수정 이력을 기록하고 필요할 때 이전 상태로 돌아갈 수 있도록 해준다.

---

##  2. Git을 사용하는 이유

###  1) 변경 사항 추적

* 파일이 언제, 어떻게 수정되었는지 기록 가능
* 이전 버전으로 언제든지 복구 가능
* 여러 파일을 따로 저장하지 않아도 됨

###  2) 협업 가능

* Github를 통해 여러 사람이 동시에 작업 가능
* 동일한 프로젝트를 각자의 컴퓨터에서 작업 가능
* 작업 내용을 공유하고 합칠 수 있음

###  3) 분산 버전 관리

* 모든 사용자가 전체 기록을 가지고 있음
* 각자 작업 후 결과를 합치는 구조
* 중앙 서버 없이도 작업 가능

---

##  3. Git 구조

###  1) Working Directory

* 실제로 작업하는 공간
* 파일을 수정하는 곳

###  2) Staging Area

* commit 전에 파일을 올려두는 공간
* 선택적으로 파일을 commit할 수 있음

###  3) Local Repository

* commit을 통해 저장되는 공간
* 변경 사항이 기록되는 곳

---

##  4. Git 기본 명령어

###  add

```bash
git add .
```

→ 변경된 파일을 staging area에 올림

---

###  commit

```bash
git commit -m "커밋 메시지"
```

→ 변경 사항을 저장

---

###  push

```bash
git push origin 브랜치이름
```

→ Github(원격 저장소)에 업로드

---

###  status

```bash
git status
```

→ 현재 파일 상태 확인

---

##  5. .gitignore

* Git이 추적하지 않을 파일을 지정하는 파일
* 보안 파일, 불필요한 파일 등을 제외할 때 사용

### 예시

```bash
ignore.md
```

또는

```bash
*.txt
```

---

##  6. Branch (브랜치)

* 독립적인 작업 공간
* 여러 기능을 동시에 개발할 수 있음
* 서로 영향을 주지 않고 작업 가능

### 브랜치 생성 및 이동

```bash
git switch -c 브랜치이름
```

---

##  7. merge

```bash
git merge 브랜치이름
```

* 다른 브랜치의 내용을 현재 브랜치에 합치는 기능
* 충돌이 발생할 수 있음

---

##  8. commit 기록 확인

```bash
git log
```

* 커밋 기록을 확인할 수 있음
* 커밋 ID, 작성자, 시간, 메시지 포함

---

##  9. HEAD

* 현재 브랜치의 최신 commit을 가리키는 포인터
* 내가 현재 작업 중인 위치를 의미

---

##  10. Issue

* 작업할 내용을 정리하는 기능
* 협업 시 할 일을 관리하는 도구

### 포함 내용

* 제목
* 작업 내용
* 담당자
* 라벨

---

##  11. Pull Request (PR)

* 작업한 내용을 main 브랜치에 반영 요청
* 코드 리뷰를 통해 병합 진행

---

##  느낀 점

Git은 단순한 파일 저장이 아니라 **버전을 관리하고 협업을 가능하게 하는 중요한 도구**라는 것을 이해했다.
특히 브랜치를 통해 독립적으로 작업할 수 있는 점이 매우 인상적이었다.
