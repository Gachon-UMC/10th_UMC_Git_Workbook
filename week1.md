# week1.md: Git 기초 및 협업 흐름 정리


##  1. Git이란 무엇인가?
* 정의: 파일의 변경 사항을 추적하고 여러 사용자 간의 작업을 조율하는 분산 버전 관리 시스템(VCS).
* 사용이유: 
    * `최종`, `진짜_최종` 등의 불필요한 파일 복사본 생성 방지.
    * 수정 내역을 로그 형식으로 관리하여 언제든 특정 시점으로 복구 가능.
    * Github를 활용해 여러 명이 동시에 효율적으로 협업 가능.

---

## 2. Git의 3가지 작업 영역
Git은 파일의 상태를 세 가지 영역으로 나누어 관리함

1. Working Directory: 실제 파일 수정이 일어나는 내 컴퓨터의 작업 폴더.
2. Staging Area: 저장(`commit`)하기 전, 반영할 파일들만 골라서 올려두는 준비 영역.
3. Local Repository: 내 컴퓨터에 버전이 영구적으로 기록되는 저장소.



---

## 3. 핵심 명령어 프로세스
기본적인 작업 흐름은 `add` → `commit` → `push` 순서로 진행됩니다.

| 명령어 | 설명 |
| :--- | :--- |
| `git clone [URL]` | 원격 저장소의 내용을 내 로컬 환경으로 복제 |
| `git status` | 현재 파일들의 상태(Modified, Staged 등) 확인 |
| `git add [파일명]` | 파일을 Staging Area에 추가 (전체 추가는 `git add .`) |
| `git commit -m "메시지"` | 변경 사항을 로컬 저장소에 기록 (체크포인트 생성) |
| `git push origin [브랜치]` | 로컬의 변경 사항을 Github(원격)에 업로드 |
| `git log` | 지금까지 만든 커밋 이력(히스토리) 확인 |

---

## 4. .gitignore 관리
* 역할: 보안상 중요한 토큰, 설정 파일 등 Git이 추적하지 말아야 할 파일 목록을 지정합니다.
* 작성법: 
    * `temp1.txt`: 특정 파일 제외
    * `*.txt`: 특정 확장자 전체 제외
    * `folder_name/`: 특정 폴더 전체 제외

---

## 5. 브랜치(Branch)와 협업 도구
* Branch: 메인 작업물에 영향을 주지 않고 독립적으로 기능을 개발하는 공간.
    * `git switch [이름]`: 브랜치 이동 (기존 `checkout`보다 권장됨).
    * `git merge [이름]`: 현재 브랜치에 다른 브랜치의 내용을 합침.
* Issue: 작업 목표(기능 구현, 버그 수정 등)를 공유하고 관리하는 게시판.
* Pull Request (PR): 작업 완료 후 내 브랜치를 메인 브랜치에 합쳐달라고 요청하는 단계. 코드 리뷰를 통해 협업 품질을 높임.
