## Git이 뭔가요

파일 저장할 때 최종, 찐최종, 찐막, 찐찐막

이렇게 변동 사항 저장할 때가 있습니다

이런 짓을 버전관리라고 하는데 Git 이라는 걸 사용하면 버전관리를 안 귀찮게 할 수 있음

**변경 사항 추적이 가능함**

- 앞에서 말한 것 처럼 파일을 여러 개 저장하면서 버전관리할 필요 없이 파일 하나에서 변동 사항 계속 저장하고 되돌리기도 하면서 버전 관리가 가능함

**Q. 근데 그럼 다같이 작업할 때는 어떡함?**

- 깃허브라는 쉽게 말하면 깃에서 쓰는 구글 드라이브 같은 게 있음
- 그래서 내 컴퓨터에서도 버전 관리가 가능한데 깃허브에 버전 관리 가능한 파일을 올려서 내용 공유하면서 협업이 가능

**분산 버전관리 가능**

- 로컬에서도 버전 관리가 가능한데 그걸 나중에 깃허브에 합칠 수도 있음
- 그래서 로컬, 원격에서 버전관리가 따로 이루어지고 나중에 공유도 돼서 편함

## 파일 버전관리 과정

내가 어떤 폴더 안에서 코딩을 하고 있다고 가정해보자

1. 현재 내가 코드 짜고 있는 파일 있는 폴더에서 작업함 (working directiory라고함)
2. 중간에 기록하기전에 `git add` 써서 원하는 파일만 대기줄에 올려 놓을 수 있음 (Staging Area)
3. 이제 기록하고 싶으면 `git commit`으로 기록 기능 (Local Repository)

## 브랜치

한 파일에서 작업하는데 상대 코드 건들여서 에러나면 좀 미안해짐

그래서 브랜치라는 걸 사용할 수 있음

- Main 브랜치: 최종적으로 코드를 합치는 찐메인 브랜치
- 기능 브랜치: A, B 이런식으로 아무렇게나 브랜치명 작명해서 서로 독립된 공간에서 개발 가능
- Merge: 각자 브랜치에서 개발했으면 이제 병합 가능

## 이슈, PR

깃허브에서 협업할때 다른사람 코드보고 잘했다고 칭찬해 줄 수도 있고, 잔소리 해야할 때도 있음

그때 이슈하고 , PR이라는 기능을 사용하면 좋음

**이슈**

- 로그인 기능 만들어야한다, 버그가 나왔다 이런거 처럼 투두리스트 처럼 사용 가능

**Pull Request(PR)**

- 독립된 브랜치에서 개발하다가 이제 메인브랜치에 합쳐야 함
- 그럴 때 PR을 작성하는데
- 내가 어떤 작업 했는지 등의 내용을 적고 팀원한테 합쳐달라고 요청하는 것
- 여기서 팀원이 내 코드보고 리뷰도 가능하고 병합 승인도 가능함

## 깃 명령어들

- `git clone`: 원격 저장소 복사
- `git status`: 현재 파일들의 상태(수정됐는지, 스테이징했는지) 확인
- `git add .`: 수정된 모든 파일을 스테이징하기
- `git commit -m "메시지"`: 스테이징한 파일을 메시지와함께 기록
- `git push origin 브랜치명`: 내 컴퓨터 기록을 GitHub로 전송
- `git switch 브랜치명`: 작업 공간(브랜치) 이동
